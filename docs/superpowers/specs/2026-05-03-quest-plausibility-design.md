# Quest Plausibility Design

**Date:** 2026-05-03
**Topic:** Fix two design issues in the quest generation system

## Problem

Two implausible quest scenarios can currently be generated by `QuestManager`:

1. **Origin/Destination overlap** — A quest may ask the player to deliver a good to a planet that itself sells the good. The player can simply buy it at the destination, defeating the delivery loop.
2. **Crafted goods, unrealistic deadlines** — Crafted goods (Circuit Board, Power Cell, Combat Module, etc.) are valid quest deliverables, but the deadline calculation does not account for `build_days * qty` plus material acquisition. Example: 2× Circuit Board (2 build days each = 4) + travel within a 6-day deadline is often impossible.

Both originate in `scripts/autoloads/quest_manager.gd`:

- `_pick_destination_planet()` filters by reachability and event-tag preference only.
- `_pick_quest_good()` selects from `EconomyManager.goods`, which since `economy_manager.gd:89` includes crafted goods.
- `_make_quest()` deadline is `randi_range(3,5) + stage_bonus + difficulty_bonus + offer_bonus`, clamped to `route_days`. No crafting-aware adjustment.

## Goals

- Quests target a destination where the good is **not** locally sold for purchase.
- Crafted-goods quests give the player a realistic chance to build, source, and deliver.
- No regression for existing non-crafted quests (Food Rations, Spice, etc.).
- No persistence migration — in-flight quests in saves keep their original terms.

## Non-Goals

- Reworking quest UI, rewards economy, or chain logic.
- Filtering crafted goods out entirely (they remain a valid quest target).
- Player-aware filtering ("does the player have this recipe unlocked"). Recipes are global once a factory slot exists.

## Approach

### 1. Helper: `EconomyManager.is_good_sold_at_planet(planet_name, good_name) -> bool`

Reads `_type_available_goods` for the planet's type and returns whether `good_name` is in the available list. Centralizes the check used by quest generation.

### 2. Helper: `CraftingManager.get_recipe_for_good(good_name) -> Resource`

Iterates `_recipes` (already loaded at `_ready()`) and returns the recipe whose `output_good.good_name` matches. Returns `null` if not crafted. Lookup runs once per quest generation, performance is non-issue.

### 3. `QuestManager._pick_destination_planet()` — strict filter

Inside the reachability loop, exclude any candidate where
`EconomyManager.is_good_sold_at_planet(candidate, good_name) == true`.

If after filtering `reachable_candidates` is empty → `return ""` (signal "no valid destination"). The caller `_make_quest()` will retry with a different good.

Existing event-tag preference logic stays — it just runs on the already-filtered list.

### 4. `QuestManager._make_quest()` — retry loop + crafting modifier

Wrap good-pick + destination-pick in a small retry loop:

```gdscript
var good: Resource = null
var dest: String = ""
for attempt in 5:
    good = _pick_quest_good(planet.planet_name, quality)
    if good == null:
        continue
    dest = _pick_destination_planet(planet.planet_name, all_planet_names, good.good_name)
    if dest != "":
        break
if good == null or dest == "":
    return {}
```

After good is picked, look up the recipe and adjust deadline + reward:

```gdscript
var recipe: Resource = CraftingManager.get_recipe_for_good(good.good_name)
if recipe != null:
    var craft_days: int = int(recipe.build_days) * qty
    var material_buffer: int = 2 if recipe.inputs.size() <= 2 else 3
    deadline += craft_days + material_buffer
    reward_mult *= 1.5
```

Reward and penalty are computed using the modified `reward_mult`, deadline as `days_left`. Penalty stays at `PENALTY_RATIO * reward`.

## Examples

| Quest | Old deadline | New deadline | Old reward | New reward |
|---|---|---|---|---|
| 2× Food Rations to Industrial planet | 3-5 + route | unchanged | unchanged | unchanged |
| 2× Circuit Board (2 inputs, 2 build_days) | 3-5 + route | + 4 craft + 2 buffer = +6 | base | × 1.5 |
| 3× Combat Module (3 inputs, 4 build_days) | 3-5 + route | + 12 craft + 3 buffer = +15 | base | × 1.5 |
| 1× Cargo Optimizer (3 inputs, 4 build_days) | 3-5 + route | + 4 craft + 3 buffer = +7 | base | × 1.5 |

## Files Touched

- `scripts/autoloads/economy_manager.gd` — add `is_good_sold_at_planet()`
- `scripts/autoloads/crafting_manager.gd` — add `get_recipe_for_good()`
- `scripts/autoloads/quest_manager.gd` — retry loop in `_make_quest()`, filter in `_pick_destination_planet()`, crafting deadline/reward modifier

## Risks & Mitigations

- **Risk:** Some goods are sold at *every* reachable planet (e.g., Raw Ore is in 3 of 5 planet types). Filter could often return empty.
  **Mitigation:** Retry loop picks a different good. With 5 attempts and non-crafted goods varying widely, near-certain to find a valid pair. Empty quest offers were already possible (e.g., Hostile faction), so the failure mode is not new.

- **Risk:** Crafted-quest reward × 1.5 could imbalance the economy.
  **Mitigation:** Crafted goods cost raw materials (player pays Electronics + Raw Ore for a Circuit Board) and tie up factory slots. The 1.5× compensates that cost; testing during play will reveal if tuning is needed.

- **Risk:** Existing quests in save files keep old deadlines.
  **Mitigation:** Acceptable — `_make_quest()` only runs for *new* offers. Old quests will resolve naturally.

## Out of Scope / Follow-ups

- Quest-UI hint that a deliverable is craftable ("requires building").
- Allowing the player to see the resolved deadline breakdown.
- Tuning the 1.5× multiplier or buffer sizes after playtesting.
